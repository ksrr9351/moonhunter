Smart Contract Architecture: Approach 1 — Fee Proxy Contract
How it works:

User Wallet → Moon Hunters Fee Proxy SC (takes 1-2% fee) → 1inch Router (actual swap) → Tokens to User
Buy Flow:

User clicks "Buy" on a dump opportunity
User approves USDT spending to Moon Hunters SC
SC receives USDT → deducts platform fee (e.g., 2%) → sends remaining to 1inch for swap
1inch swaps USDT → target token → token goes directly to user's wallet
Fee goes to Moon Hunters treasury wallet
Sell Flow:

User clicks "Sell" on their position
User approves token spending to Moon Hunters SC
SC receives token → sends to 1inch for swap → gets USDT back
SC deducts fee from USDT → sends remaining to user's wallet
Fee goes to treasury
SC Key Functions:

buyToken(address token, uint256 usdtAmount, uint256 minOutput, bytes swapData) — Buy via 1inch with fee + slippage protection
sellToken(address token, uint256 tokenAmount, uint256 minUsdtOutput, bytes swapData) — Sell via 1inch with fee + slippage protection
setFeePercent(uint256 fee) — Owner adjusts fee (max 5% hardcoded cap)
withdrawFees() — Transfer collected fees to treasury
pause() / unpause() — Emergency stop
recoverTokens(address token, uint256 amount) — Recover accidentally sent tokens
addWhitelistedToken(address) / removeWhitelistedToken(address) — Optional token whitelist
SC Security Features:

ReentrancyGuard (OpenZeppelin) on buy/sell
Slippage protection via minOutputAmount
Max fee cap (5%) hardcoded — owner cannot exceed
Event logging (BuyExecuted, SellExecuted, FeeCollected, FeeUpdated, Paused/Unpaused)
Emergency token recovery
Optional token whitelist
Why 1inch (not own liquidity):

No capital needed for liquidity pools
200+ DEXes aggregated = best price always
Any token support — whatever dumps, it's tradeable
Battle-tested, handles millions of trades daily
Multi-chain deployment: Same SC code deployed on ETH, Polygon, Arbitrum, BSC — each chain has its own SC address + 1inch router.

SC Security Features (to be included in Solidity contract):
1. Reentrancy Protection: Use OpenZeppelin ReentrancyGuard on buyToken/sellToken — prevents attack between external calls
2. Slippage Protection: minOutputAmount parameter on both buy and sell — user gets guaranteed minimum tokens, prevents front-running/sandwich attacks
3. Max Fee Cap: Hardcoded MAX_FEE = 500 (5%) in SC — setFeePercent() reverts if fee > MAX_FEE, even owner cannot exceed this, builds user trust
4. Token Whitelist (Optional): addWhitelistedToken(address) / removeWhitelistedToken(address) — owner can optionally restrict to verified tokens only, whitelistEnabled toggle (default: off, any token allowed)
5. Event Logging: Emit events on every action for transparency and frontend tracking:
event BuyExecuted(address user, address token, uint256 amountIn, uint256 amountOut, uint256 fee)
event SellExecuted(address user, address token, uint256 amountIn, uint256 amountOut, uint256 fee)
event FeeCollected(address token, uint256 amount)
event FeeUpdated(uint256 oldFee, uint256 newFee)
event Paused(address by) / event Unpaused(address by)
6. Emergency Token Recovery: recoverTokens(address token, uint256 amount) — owner-only function to recover tokens accidentally sent directly to SC (not from trades), prevents permanent loss of user funds
Add event listeners in contractService.js to track BuyExecuted/SellExecuted events for real-time UI updates
